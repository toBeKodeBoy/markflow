## 3. v1.1 详细计划：组织与检索

> **主题：** 让笔记「找得到、排得顺、标得清」

### 3.1 全文搜索增强

**现状：** 标题 + 正文已可搜（`contentSearchIndex`）；缺标签匹配、摘要展示与防抖。

**目标：** 支持标题 + 正文 + 标签联合搜索，结果高亮匹配片段。

#### 任务分解

| # | 任务 | 文件 | 估时 |
|---|------|------|------|
| 1.1.1 | 新增 `searchIndex` 缓存结构（id → 可搜索文本） | `src/stores/note.ts` | ✅ |
| 1.1.2 | 笔记保存/更新时同步索引 | `note.ts` | ✅ |
| 1.1.3 | 扩展 `filteredNoteList` 支持 content/tags 匹配 | `note.ts` | 🔄 缺 tags |
| 1.1.4 | 搜索结果展示匹配摘要（首段命中上下文） | `Sidebar.vue` | 1d |
| 1.1.5 | 搜索防抖（300ms）与空状态 UI | `Sidebar.vue` | 0.5d |
| 1.1.6 | 单元测试：标题/正文/标签/组合搜索 | `tests/unit/stores/note.test.ts` | 0.5d |

#### 技术方案

```typescript
// 索引策略：内存 Map，启动时懒加载
interface SearchIndexEntry {
  title: string
  contentLower: string  // 全文小写缓存
  tags: string[]
}

// 笔记 > 500 篇时考虑 Web Worker 构建索引
```

**验收标准：**
- 输入关键词可匹配笔记正文内容
- 500 篇笔记下搜索响应 < 100ms
- 文件夹过滤与搜索可叠加

### 3.3 笔记排序

**现状：** 列表按 `updatedAt` 倒序，无自定义排序。

#### 任务分解

| # | 任务 | 文件 | 估时 |
|---|------|------|------|
| 1.3.1 | `NoteListItem` 扩展 `pinned?: boolean` 和 `sortOrder?: number` | `types/index.ts` | 0.5d |
| 1.3.2 | 排序逻辑：置顶 > sortOrder > updatedAt | `note.ts` | 0.5d |
| 1.3.3 | 右键菜单：置顶/取消置顶 | `Sidebar.vue` | 0.5d |
| 1.3.4 | 拖拽排序（HTML5 DnD 或轻量 composable） | `Sidebar.vue` + `useNoteSort.ts`（新） | 1.5d |
| 1.3.5 | 文件夹内排序 vs 全局排序策略确定 | 设计决策 | 0.5d |
| 1.3.6 | 数据迁移：旧列表补全默认 sortOrder | `useStorage.ts` | 0.5d |
| 1.3.7 | 测试 | `tests/` | 0.5d |

#### 排序规则

```
filteredNoteList 排序优先级：
1. pinned === true 的笔记在最前（pinned 内部按 updatedAt 倒序）
2. 其余按 sortOrder 升序（未设置则按 updatedAt 倒序）
```

**验收标准：**
- 置顶笔记始终显示在列表顶部
- 拖拽后顺序持久化，重启后保持

## 5. v2.0 规划概要：平台化演进

> 完整说明见 **[v2.0.md](./v2.0.md)**。前置条件：v1.2 稳定运行 ≥ 1 个月，架构评审通过。

### 5.1 云同步

| 方案 | 优点 | 缺点 |
|------|------|------|
| WebDAV（坚果云/Nextcloud） | 用户自控、实现简单 | 需用户配置 |
| uTools 账号同步 | 零配置 | 依赖平台能力 |
| 自建后端 | 完全可控 | 运维成本 |

**建议路径：** 先做 WebDAV 同步 → 评估 uTools 官方能力 → 再考虑自建。

### 5.2 多 Tab 编辑

- 改造 `currentNote` 为 `openTabs: NoteTab[]` + `activeTabId`
- 未保存提示（切换 Tab / 关闭时）
- 内存管理：超过 5 个 Tab 时卸载非活跃编辑器实例

### 5.3 插件扩展

- 定义扩展点：渲染器插件、导出插件、快捷键插件
- 参考 Obsidian 社区插件模式的简化版
- **风险高，建议在 v2.0 后期迭代**
